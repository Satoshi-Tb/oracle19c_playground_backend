ORA-01461 MyBatis CLOBバインド問題

RESOLVED Oracle JDBC + MyBatis ClobTypeHandler — 発生原因と OracleClobStringTypeHandler による解決

1

概要

MyBatisで jdbcType=CLOB を指定していたにもかかわらず、Oracle JDBC側でバインド変数が CLOBではなくLONG系として解釈され、CLOB連結のUPDATEでエラーになりました。 問題が発生したSQLは LobMapper.xml の updateDetail です。

更新時の fragment は MyBatisLobService で40,000文字の文字列として生成されます。 期待した状態は「fragment をCLOBとしてバインドする」ことでしたが、 MyBatis標準の ClobTypeHandler が使用するJDBC APIとOracle JDBCの型解釈の組み合わせにより、 実際にはLONG系として扱われました。

LobMapper.xml — 問題のSQL断片 
FRAGMENT = FRAGMENT || TO_CLOB('=>') ||
    #{fragment,jdbcType=CLOB}
発生エラー: ORA-01461: LONG値はLONG列にのみバインドできます
原因はMyBatis標準 ClobTypeHandler が PreparedStatement#setCharacterStream を使用するため、 Oracle JDBCがCLOB連結式の ? をLONG系として解釈したことです。
重要なポイント: jdbcType=CLOB を指定した時点でOracleにCLOBとして渡ることが保証されるわけではありません。 MyBatisはその指定をもとに TypeHandlerを選択 しますが、実際にどのJDBC APIで値を設定するかは TypeHandlerの実装に依存します。

3

MyBatisがパラメータを解釈する順番

Mapper XMLの #{...} はそのままJDBCに渡されるわけではありません。実行前に ParameterMapping として解析され、実行時に TypeHandler を経由して PreparedStatement に値が設定されます。

  1. Mapper XMLの #{fragment,jdbcType=CLOB} を解析する
  2. fragment のJava型を確認する → String
  3. XMLで指定されたJDBC型を確認する → CLOB
  4. typeHandler が明示指定されていなければ、Java型とJDBC型から標準TypeHandlerを選択する
  5. String + JdbcType.CLOB の組み合わせでは標準の ClobTypeHandler が使用される
  6. 実行時に DefaultParameterHandler がSQLパラメータを順番に処理する
  7. 各パラメータについて、選択済みのTypeHandlerが PreparedStatement に値を設定する
  8. 最終的なバインド型は、呼び出されたJDBC APIをもとに JDBCドライバが解釈 する ← ここで問題が発生
4

修正前の挙動

修正前は String + jdbcType=CLOB に対してMyBatis標準の ClobTypeHandler が選択されていました。 このハンドラは PreparedStatement#setCharacterStream を使用しますが、Oracle JDBCでは CLOB連結式の ? がCLOBとして確定せず、LONG系として解釈されてエラーになりました。

MyBatis 標準 ClobTypeHandler（該当箇所抜粋） 
@Override
public void setNonNullParameter(PreparedStatement ps, int i, String parameter, JdbcType jdbcType)
    throws SQLException {
  StringReader reader = new StringReader(parameter);
  ps.setCharacterStream(i, reader, parameter.length());
}
なぜ setCharacterStream がLONG系になるのか: setCharacterStream はJDBC標準APIで文字ストリームを渡しますが、Oracle JDBCでは SQL式や値の長さによってLONG系として扱われるケースがあります。 今回の FRAGMENT || TO_CLOB('=>') || ? という連結式で ? がCLOBとして確定しませんでした。

段階 状態 問題の有無
Mapper XML解析 jdbcType=CLOB は認識されている 問題なし

解決方法

Oracle JDBCの場合だけ OraclePreparedStatement#setStringForClob を使用する独自TypeHandlerを追加しました。 ClobTypeHandler を継承し、書き込み時の setNonNullParameter だけをオーバーライドしています。 Oracle JDBCとしてunwrapできない場合は、標準の ClobTypeHandler に委譲します。

OracleClobStringTypeHandler.java 
@MappedTypes(String.class)
@MappedJdbcTypes(JdbcType.CLOB)
public class OracleClobStringTypeHandler extends ClobTypeHandler {

    @Override
    public void setNonNullParameter(PreparedStatement ps, int i, String parameter, JdbcType jdbcType)
            throws SQLException {
        if (ps.isWrapperFor(OraclePreparedStatement.class)) {
            ps.unwrap(OraclePreparedStatement.class).setStringForClob(i, parameter);
            return;
        }

        super.setNonNullParameter(ps, i, parameter, jdbcType);
    }
}
LobMapper.xml — 修正後の局所適用 
#{fragment,jdbcType=CLOB,typeHandler=com.example.dbperf.typehandler.OracleClobStringTypeHandler}
修正後 OracleClobStringTypeHandler → setStringForClob → 成功

設計判断の理由

局所適用にした理由

  • 問題が確認されたのはCLOB連結式を含む updateDetail のみ。CLOB列への通常INSERTや単純UPDATEでは発生していない。
  • 影響範囲を最小化するため、Mapper XMLで局所的にTypeHandlerを指定している。
  • 全体適用する場合は application.properties の mybatis.type-handlers-package で登録可能だが、String + CLOB の全箇所に影響するため現状は局所適用を継続。

TypeHandlerを全て一から自作しなかった理由

  • 差し替えたかった処理は、CLOB値を書き込むときの バインドAPI選択だけ。
  • ClobTypeHandler を継承し setNonNullParameter だけをオーバーライドすることで、読み取り処理や非Oracle環境の標準処理を再利用できる。
  • Oracle JDBCとしてunwrapできる場合のみ問題のAPIを差し替え、できない場合は標準実装に委譲するため、独自実装範囲が明確で小さい。
  • @MappedTypes と @MappedJdbcTypes はパッケージスキャン登録時のメタ情報。Mapper XMLで明示指定している現状ではそちらが優先される。

Reader型へ変更しなかった理由

  • ClobReaderTypeHandler を使う場合、DTOの fragment フィールドを String から Reader に変更する必要がある。
  • Reader はストリームであり現在位置を持つため、一度読み進めると同じインスタンスを再利用できない。
  • DTOが「値の入れ物」として扱いにくくなり、ログ出力・テスト値比較・レスポンス検証などで不便が生じる。
  • 問題はJava側の表現型ではなく Oracle JDBCへのバインドAPI選択 にあるため、モデルを変更せずTypeHandlerでバインド方法だけを変更した。
