DBPerfSample Technical Note

MyBatis CLOBバインド問題の整理

`jdbcType=CLOB` を指定していたにもかかわらず、Oracle JDBC側でバインド変数がLONG系として解釈された問題と、 `OraclePreparedStatement#setStringForClob` を使った最終対応を整理したHTML版ドキュメントです。

Symptom ORA-01461
Trigger CLOB連結式に40,000文字のStringをbind
Fix OracleClobStringTypeHandlerを局所適用
Overview

概要

何が起きたか

MyBatisで jdbcType=CLOB を指定していたにもかかわらず、 Oracle JDBC側でバインド変数がCLOBではなくLONG系として解釈されました。 その結果、CLOB連結のUPDATEで ORA-01461 が発生しました。

期待していた状態

fragment をCLOBとしてバインドし、 FRAGMENT || TO_CLOB('=>') || ? の ? をCLOB連結式の一部として正しく扱わせることです。

src/main/resources/mapper/LobMapper.xml 修正前の問題SQL断片 
FRAGMENT = FRAGMENT || TO_CLOB('=>') ||
    #{fragment,jdbcType=CLOB}
問題の本質

Mapper XML上のCLOB指定が不足していたわけではありません。 MyBatisはCLOB指定を認識していましたが、標準CLOBハンドラが呼び出すJDBC APIとOracle JDBCの解釈が、今回のSQL式には合いませんでした。

Parameter Mapping

MyBatisがパラメータを解釈する順番

Mapper XMLの #{...} は、そのままJDBCに渡されるのではなく、実行前に ParameterMapping として解析され、実行時に TypeHandler を経由して PreparedStatement に設定されます。

1 Mapper XML解析 #{fragment,jdbcType=CLOB} を解析する。
2 Java型確認 fragment のJava型は String。
3 JDBC型確認 XML指定によりJDBC型は CLOB。
4 Handler選択 明示指定がなければ型の組み合わせから標準Handlerを選ぶ。
5 標準Handler String + CLOB では ClobTypeHandler が選択される。
6 実行時設定 DefaultParameterHandler がSQL上の ? の順に処理する。
7 JDBC API呼出 TypeHandlerが PreparedStatement に値を設定する。
8 Driver解釈 最終的なbind型は呼び出されたJDBC APIをもとにドライバが解釈する。
Before Fix

修正前のシーケンス

修正前は、String + jdbcType=CLOB に対してMyBatis標準の ClobTypeHandler が使われていました。この標準ハンドラは PreparedStatement#setCharacterStream を使用します。

setCharacterStream は文字ストリームを渡すJDBC標準APIですが、Oracle JDBCではSQL式や値の長さによってLONG系として扱われるケースがあります。 今回はCLOB連結式の ? がCLOBとして確定せず、LONG系として解釈されたことが直接原因です。

Fault Boundary

どこで型の解釈エラーが発生したか

型解釈のズレは、MyBatisの ParameterMapping 作成時ではなく、 TypeHandlerがJDBCドライバへ値を渡す段階からOracle実行時にかけて発生しました。

段階 状態 問題の有無
Mapper XML解析 jdbcType=CLOB は認識されている 問題なし
TypeHandler選択 String + CLOB から ClobTypeHandler を選択 標準動作として妥当
TypeHandler実行 setCharacterStream で値を設定 OracleではLONG系解釈の余地
Oracle実行 CLOB連結式の ? がLONG系として扱われる ここでORA-01461

MyBatis既存実装のコードフラグメント

以下は、MyBatis 3.5.19 の実装から、今回の経路を理解するために必要な部分だけを抜粋したものです。 jdbcType=CLOB により ClobTypeHandler は選ばれていますが、 最終的にOracle JDBCへ渡るAPIは setCharacterStream です。

TypeHandlerRegistry String + CLOB の登録 
register(String.class, JdbcType.CLOB, new ClobTypeHandler());
DefaultParameterHandler 実行時の呼び出し 
TypeHandler typeHandler = parameterMapping.getTypeHandler();
JdbcType jdbcType = parameterMapping.getJdbcType();
typeHandler.setParameter(ps, i + 1, value, jdbcType);
ClobTypeHandler CLOB値の設定 
ps.setCharacterStream(i, new StringReader(parameter), parameter.length());

この実装から、今回の問題はMyBatisがCLOB指定を認識しなかったことではなく、 標準 ClobTypeHandler のJDBC API選択がOracleのCLOB連結式ではLONG系解釈につながったことだと整理できます。

Final Solution

最終的な解決方法

Oracle JDBCの場合だけ OraclePreparedStatement#setStringForClob を使用する独自TypeHandlerを追加しました。 これにより、問題のSQL式に渡す値をOracle JDBCへ明示的にCLOBとしてバインドします。

src/main/java/com/example/dbperf/typehandler/OracleClobStringTypeHandler.java 要点 
@MappedTypes(String.class)
@MappedJdbcTypes(JdbcType.CLOB)
public class OracleClobStringTypeHandler extends ClobTypeHandler {

    @Override
    public void setNonNullParameter(PreparedStatement ps, int i, String parameter, JdbcType jdbcType)
            throws SQLException {
        if (ps.isWrapperFor(OraclePreparedStatement.class)) {
            ps.unwrap(OraclePreparedStatement.class).setStringForClob(i, parameter);
            return;
        }

        super.setNonNullParameter(ps, i, parameter, jdbcType);
    }
}

Oracle JDBCの場合

isWrapperFor と unwrap で OraclePreparedStatement に到達し、setStringForClob を呼びます。

unwrapできない場合

super.setNonNullParameter(...) に委譲し、MyBatis標準の ClobTypeHandler の動作を利用します。

After Fix

修正後のシーケンス

修正後は、問題が発生した updateDetail の fragment に 独自TypeHandlerを明示指定しています。

src/main/resources/mapper/LobMapper.xml 局所指定

Scope

局所適用にした理由

問題が確認されたのは、CLOB列への通常INSERTや単純UPDATEではなく、CLOB連結式を含む updateDetail でした。そのため、影響範囲を絞るためにMapper XMLで局所的にTypeHandlerを指定しています。

局所適用

問題が出た updateDetail.fragment だけを差し替える。

instead of

全体適用

String + CLOB の全箇所に影響するため、今回は採用しない。

application.properties 全体適用する場合の代替手段 
# mybatis.type-handlers-package=com.example.dbperf.typehandler

@MappedTypes(String.class) と @MappedJdbcTypes(JdbcType.CLOB) は、 TypeHandlerをパッケージスキャンなどで登録する場合に、どのJava型・JDBC型に対応するハンドラかをMyBatisへ伝えるためのメタ情報です。

Implementation Choice

TypeHandlerを全て一から自作しなかった理由

今回差し替えたかった処理は、CLOB値を書き込むときのバインド方法だけです。 具体的には、MyBatis標準の ClobTypeHandler が setCharacterStream を呼ぶ部分を、 Oracle JDBCの場合だけ OraclePreparedStatement#setStringForClob に置き換えたい、という限定的な要件でした。

再利用

MyBatis標準のCLOB処理をできるだけ再利用できます。

差分を限定

読み取り処理など、今回の問題と関係しない処理を独自実装しなくて済みます。

fallback

Oracle JDBCとしてunwrapできない場合は、標準の ClobTypeHandler に委譲できます。

記録しておく判断

このTypeHandlerはCLOB処理を全面的に作り直すためのクラスではありません。 標準 ClobTypeHandler のうち、Oracleで問題になったバインド処理だけを差し替えるためのクラスです。

Alternative

Reader型へ変更しなかった理由

MyBatisには Reader をCLOBとして扱うための ClobReaderTypeHandler もあります。 しかし、その場合はDTOやモデルの fragment を String ではなく Reader として扱う必要が出ます。

LobDetail.java Reader型サンプル 
public class LobDetail {

    private Long id;
    private Long headerId;
    private Long no;
    private Reader fragment;
    private Long fragmentLen;
    private String memo;
}
MyBatisLobService.java StringReader設定サンプル 
String updatedDetailText = createSampleText("UPDATE_SAMPLE:", 40000);

LobDetail detail = new LobDetail();
detail.setId(insertResult.getDetailId());
detail.setNo(1L);
detail.setFragment(new StringReader(updatedDetailText));
detail.setFragmentLen((long) updatedDetailText.length());
detail.setMemo("MyBatis updated sample data");
LobMapper.xml javaType明示例 
#{fragment,jdbcType=CLOB,javaType=java.io.Reader}

ストリームとしての制約

Reader は値そのものではなく、値を読み出すためのストリームです。 一度読み進めると、同じインスタンスをそのまま再利用できません。

DTOとしての利便性

ログ出力、テストでの値比較、レスポンス検証、同じ値を複数回使う処理では、 中身を直接保持する String のほうが扱いやすいです。

今回のアプリでは、fragment はサービス層・テスト・レスポンス確認で文字列として扱うほうが自然です。 問題はJava側の表現型ではなく、Oracle JDBCへのバインドAPI選択にありました。 そのため、モデルを Reader に寄せるのではなく、String のままOracle向けTypeHandlerでバインド方法だけを変更しました。
